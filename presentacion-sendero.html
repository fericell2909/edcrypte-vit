08 — Webhooks (Critico)

Recepcion de Eventos

Los webhooks son la fuente oficial de estado de pagos. Endpoint: POST /api/webhooks/toku

Eventos Esperados

payment.succeeded
Pago procesado correctamente
payment.failed
Pago rechazado o fallido
subscription.updated
Suscripcion actualizada
subscription.canceled
Suscripcion cancelada

Validaciones Requeridas

Firma del Webhook

Validacion criptografica de la firma usando TOKU_WEBHOOK_SECRET para asegurar autenticidad.

Idempotencia

Prevencion de procesamiento duplicado de eventos mediante registro de IDs unicos.

Registro de Eventos

Almacenamiento completo de cada evento recibido para auditoria y trazabilidad.

09 — Estados

Manejo de Estados de Pago

Maquina de estados para el ciclo de vida de cada pago, con historial completo para auditoria.

Historial de Eventos

Se almacenara un log completo de transiciones de estado con timestamp, evento origen y metadata para garantizar trazabilidad total y facilitar la conciliacion.

12 — Riesgos

Riesgos Tecnicos

Riesgos identificados y su impacto potencial en la integracion.

Webhooks Incorrectos

Si los webhooks no se validan correctamente, se pueden generar estados inconsistentes entre la plataforma y Toku.

Impacto: Estados inconsistentes

Falta de Idempotencia

Sin control de idempotencia, eventos duplicados pueden generar cobros duplicados a los usuarios.

Impacto: Cobros duplicados

Timeout del Gateway

Timeouts en la comunicacion con Toku pueden dejar pagos en estado inconcluso sin respuesta al usuario.

Impacto: Pagos inconclusos
13 — Buenas Practicas

Recomendaciones

✓

Guardar siempre el external_id de Toku para cada entidad creada en la API externa.

✓

Usar Webhooks como fuente de verdad para el estado de pagos y suscripciones.

✓

Registrar todos los eventos recibidos para auditoria completa y trazabilidad.

✓

No almacenar credenciales en codigo fuente. Usar variables de entorno exclusivamente.
